# MSXPLAYer Game Cartidge Adapter GUI 仕様メモ

## 目的

MSXPLAYer Game Cartidge Adapter を USB CDC のシリアルデバイスとして扱い、Python + `tkinter` で操作用 GUI を実装する。

この文書は、最初の実装方針と画面構成、通信の考え方を整理するためのもの。

## 前提

- Adapter は USB 接続時に CDC として認識され、シリアルポート経由で通信する
- 通信コマンド仕様は `docs/com_command.md` を正とする
- GUI は `tkinter` で実装する
- 外部ライブラリは最小限に抑える
- シリアル通信の既定値は `115200 / 8N1 / timeout 1秒` とする

## 画面の見た目

画面は HEX エディタに近い見た目とする。

- 左列: アドレス
- 中央列: HEX データ
- 右列: キャラクタ表示

表示は 64kB 全体を対象にし、通常の HEX エディタと同様に 1 行ごとに固定幅で並べる。
HEX の横幅は 16 バイト固定とする。
キャラクタ表示は印字可能 ASCII のみそのまま表示し、それ以外は `.` とする。
Flat 表示時のアドレスは 4 桁とする。
Mapper 表示時のアドレスは `FullAddr(6桁):Bank(2桁):Ofst(4桁)` とする。

## 表示モードの用語

- 現在の `0000h` から `FFFFh` までをそのまま表示するモードを `Flat` と呼ぶ
- マッパーによるバンク切り替えを考慮した ROM 表示モードを `Mapper` と呼ぶ

## 起動オプション

シリアルデバイスはコマンドライン引数で指定できるようにする。

例:

```text
--device=/dev/ttyACM0
```

`--device` を指定した場合は、起動時にそのデバイスへ自動接続を試みる。

## 起動時フロー

起動直後はシリアル未接続の状態で開始する。

- HEX 画面は 64kB 全体を `00` で初期表示する
- 接続は `設定 -> シリアル` またはツールバーのシリアルボタンから行う

シリアルデバイスを開いた直後は以下の順で処理する。

### 1. `HVER` 実行

- Adapter に対して `HVER` を送信する
- バージョン情報が取得できることを確認する
- 取得できなかった場合はエラー表示を出し、未接続状態に戻る

### 2. 64kB 全体の読出し

- 64kB 全体をシリアル経由で読み出す
- 読出し対象はバッファ全体の `0000h` から `FFFFh` までとする
- 初回読出し時は以下のシーケンスで行う

```text
SPON
1 秒待つ
SMTR,0000,10000,0,1
BSND,
```

- `SMTR` で Adapter 側バッファに転送し、その後 `BSND` で PC へ受信する
- `10000` は 64kB 全体を意味する長さとして扱う
- 起動後はスロット電源を ON のまま維持する
- 読出し完了まで GUI は待機状態になる

### 3. HEX 画面へ表示

- 読み出した 64kB の内容を HEX 画面へ反映する
- アドレス、HEX データ、キャラクタ表示を同時に更新する
- 読出し前の空画面ではなく、ロード進行が分かる表示を用意することが望ましい

## 推奨構成

### 必須

- Python 3.10 以上
- `tkinter`
- `pyserial`

### 推奨理由

- `tkinter` は標準ライブラリなので GUI 追加コストが低い
- シリアル通信は標準ライブラリだけでは扱いづらいため、`pyserial` の採用が最も実用的
- 依存を増やさずに、Windows / macOS / Linux のクロスプラットフォーム対応がしやすい

### 追加候補

- `pyinstaller`
  - 配布用の単体実行ファイルを作る段階で必要なら採用する
- `logging`
  - 標準ライブラリで十分。専用ライブラリは不要

## GUI の基本方針

GUI は HEX エディタのような見た目にする。
上部にメニューバーとツールバーを配置する。

### メニュー

- ファイル
  - 保存
  - 終了
- 設定
  - シリアル
- デバッグ
  - 手動実行
  - ログ

### ツールバー

- 保存
- シリアル
- 右端に `00`, `40`, `80`, `C0`, `FF` のジャンプボタンを配置する
- 各ボタンはそれぞれ `0000`, `4000`, `8000`, `C000`, `FFFF` へスクロールする

### Mapper バー

- ツールバーと HEX 表示の間に 1 行の `mapper バー` を配置する
- mapper バーには以下を配置する
  - `Mapper` 固定表示のチェックボタン
  - `Type:` ラベル
  - `MapperType` を選択するドロップダウンリスト
  - `Window:` ラベル
  - `Window` を選択するドロップダウンリスト
  - `Switch:` ラベル
  - `SwitchAddr` を選択する編集可能なドロップダウンリスト
  - `Bank:` ラベル
  - `Bank` を選択する 16 進表示のスピンボックス
  - `Map` ボタン
- `Mapper` チェックボタンは、`Off` のとき `Flat`、`On` のとき `Mapper` の動作とする
- `Flat` 時は `MapperType`、`Window`、`SwitchAddr`、`Bank` をグレーアウトして変更不可にする
- `Mapper` 時は `MapperType`、`Window`、`SwitchAddr`、`Bank` を有効にする
- `Flat` / `Mapper` を切り替えたら、対象モードのダンプを再取得して表示する
- Mapper ダンプ中は進捗表示用のプログレスバーをステータスバーに表示する

### Mapper Map ダイアログ

- `Map` ボタンを押すと Mapper Map ダイアログを開く
- ダイアログ上部に `Canvas` による `16 x 16` の表を配置する
- 上部の列ヘッダは `+0` から `+F` とする
- 左側の行ヘッダは `+00` から `+F0` とする
- マトリクスの各マスには `00` から `FF` を 16 進で表示する
- 並び順は左上から右へ `00`, `01`, ... と進み、右下を `FF` とする
- 各セルの境界線を描画する
- 初期状態は `白背景 / 黒文字` とする
- ダイアログ下部に `Analyze` ボタンを配置する
- ダイアログ下部の `Analyze` ボタンと同列左側に、解析後のイメージサイズ表示を配置する
- イメージサイズ表示は左揃えとする
- セル部分をクリックした場合はダイアログは閉じず、背後のメインウインドウで該当バンクの先頭へカーソル移動する

### Mapper Map 解析処理

- `Analyze` ボタン押下で解析を開始する
- 解析中は `00` から `FF` まで各バンクを 1 つずつ読み出してハッシュを計算する
- 解析中の進捗はステータスバーのプログレスバーで表示する

解析手順は以下とする。

1. `00` から `FF` まで 1 バンクずつ読み出してハッシュを計算する
2. まだ存在しないハッシュなら新規データとして扱う
3. 新規データだった場合、該当マスは `XX` 表示のまま `赤文字` にする
4. 既出のハッシュだった場合は、最初に一致したバンクを `before` として、マスの表示を `(YY)` にし、`グレー文字` にする

解析完了後のイメージサイズは以下で計算する。

- `bank 0` から連続した新規データ数 × バンクサイズ
- 表示は 10 進数で `XXX Bank * NkB = XXX kByte` の形式とする
5. `FF` まで繰り返す

`Window` の候補は以下の 4 種とする。

- `4000-5FFF(8k)`
- `4000-7FFF(16k)`
- `8000-9FFF(8k)`
- `8000-BFFF(16k)`

`SwitchAddr` の候補は以下とする。

- `6000`
- `6800`
- `7000`
- `7800`
- `8000`
- `8800`
- `9000`
- `9800`

`MapperType` の候補は以下とする。

- `ASCII 8K`
  - `Window: 4000-5FFF`
  - `SwitchAddr: 6000`
  - 既定値
- `ASCII 16K`
  - `Window: 4000-7FFF`
  - `SwitchAddr: 6000`
- `Konami`
  - `Window: 4000-5FFF`
  - `SwitchAddr: 5000`
- `Custom`
  - `Window: 4000-5FFF`
  - `SwitchAddr: 6000`

`Window` または `SwitchAddr` を手動変更した場合は、`MapperType` を `Custom` に切り替える。

### シリアル選択

- `設定 -> シリアル` とツールバーの `シリアル` は、シリアルデバイス選択ダイアログを開く
- ダイアログには `pyserial` により検出したシリアルデバイス一覧をドロップダウンリストで表示する
- 一覧から 1 つ選択して開けるようにする
- 現在使用中のデバイスが一覧にあれば初期選択する
- シリアルデバイスを開いた直後に初期化シーケンスを実行する
- デバイス切替時は新しい接続で `HVER` と 64kB 再読込を行い、成功時だけ切り替える
- 未保存の編集内容がある場合は、切替前に確認を出す

### 直接書き込み

- HEX 編集で 1 バイト確定時に即座に 1 バイト書き込みを行う
- 1 バイト書き込み後は `SMTR` と `BSND` により 64kB 全体を再読込し、HEX 画面を再表示する
- 直接書き込み後の再読込では `SPON`、待機、`SPOFF` は行わない
- アプリ終了時は、接続済みなら `SPOFF` を実行してからシリアルを閉じる

### 保存ダイアログ

- `保存` を押すと保存ダイアログを表示する
- ファイル名を指定できる
- 既定の保存ファイル拡張子は `.rom` とする
- `Flat` / `Mapper` のラジオボタンを配置する
- `Flat` / `Mapper` の初期値は、メインウインドウの mapper チェックボックス状態に合わせる
- ラジオボタンの下に `Flat` フレームと `Mapper` フレームを配置する
- 選択したラジオボタンに応じて、対応するフレームだけ有効にする

#### Flat フレーム

- 保存するアドレス範囲を指定できる
- 開始アドレスはテキスト入力可能なドロップダウンとする
- 終了アドレスはテキスト入力可能なドロップダウンとする
- 開始アドレス候補は `0000`, `4000`, `8000`, `C000` とし、既定値は `0000`
- 終了アドレス候補は `3FFF`, `7FFF`, `BFFF`, `FFFF` とし、既定値は `FFFF`
- 保存時は指定範囲を含むデータだけをファイルへ書き出す

#### Mapper フレーム

- `Type`, `Window`, `Switch` の入力を配置する
- これらの初期値はメインウインドウの mapper バーと同じ値にする
- `Type` を変更した場合は、対応する `Window` と `Switch` のプリセット値を反映する
- `Window` または `Switch` を手動変更した場合は `Type=Custom` に切り替える
- `StartBank`, `EndBank` を配置する
- `StartBank` の既定値は `00` とする
- `EndBank` の既定値は通常 `FF` とする
- 一度でも `Analyze` を実行したことがある場合は、最後に検知したサイズに応じて `EndBank` の既定値を変更する
- Mapper 保存時は、指定した `Type`, `Window`, `Switch`, `StartBank`, `EndBank` に基づいてバンク内容を読み出して保存する

### 保存時の入力検証

- 開始アドレスと終了アドレスは 16 進数のみ受け付ける
- `0000` から `FFFF` の範囲外はエラーとする
- 開始アドレスが終了アドレスを超えていたらエラーとする
- `Mapper` 保存時は `Switch`, `StartBank`, `EndBank` を 16 進数で受け付ける
- `StartBank` が `EndBank` を超えていたらエラーとする

## 画面構成案

### 1. HEX 表示エリア

- アドレス列
- HEX データ列
- 文字表示列
- HEX 表示エリア上部に固定ルーラを表示する
- 初期段階では閲覧専用とする
- 1 行 16 バイト固定

表示形式の基本イメージ:

- 1 行 16 バイト
- 左に 4 桁アドレス
- 中央に 16 バイトの HEX
- 右に ASCII 相当の可視文字
- 最上部に `ADDR  +0 +1 ... +F  0123456789ABCDEF` のルーラを常時表示する
- ルーラは縦スクロールの影響を受けず、横スクロール時は HEX 本体と同期する

キー入力やカーソル移動イベントをフックし、アドレスやスペースにカーソルが移動できないようにする。
HEX 部分では `0-9`, `a-f`, `A-F` 以外のキャラクタ入力は無視する。
変更した箇所は赤字にする。
- `PageUp` / `PageDown` でそれぞれ `-0100h` / `+0100h` 移動する
- `Ctrl+PageUp` / `Ctrl+PageDown` でそれぞれ `-0800h` / `+0800h` 移動する
- `Mapper` 時はカーソルキー、`PageUp` / `PageDown`、`Ctrl+PageUp` / `Ctrl+PageDown` により Bank 境界をまたいで移動できるようにする
- `Mapper` 時は、現在の表示範囲末尾を越える移動では次の Bank を読み込んで対応オフセットへ移動する
- `Mapper` 時は、現在の表示範囲先頭を越える移動では前の Bank を読み込んで対応オフセットへ移動する
- `Mapper` 時の `PageUp` / `Ctrl+PageUp` は前方向、`PageDown` / `Ctrl+PageDown` は次方向へ移動する
- `Bank=00` から前方向、`Bank=FF` から次方向への移動は行わない

書き込み操作をするまではオンメモリで編集するだけに留める。

`Flat` 時は通常の `0000h` から `FFFFh` の表示を行う。
`Mapper` 時は mapper バーの設定に従って取得したダンプを表示する。
`Mapper` 時も HEX 編集を許可する。

`Mapper` 時のアドレス欄は以下の形式で表示する。

- `FullAddr(6桁):Bank(2桁):Ofst(4桁)`

`Mapper` 時のルーラ表記は以下とする。

- `FulAdr:Bk:Ofst`

### 1-1. Mapper ダンプ手順

- バンク番号は `00` から `FF` まで扱う
- HEX 表示サイズは `Window` のサイズとする
- `Bank` スピンボックスで現在表示するバンクを選択する
- `Mapper` 時のダンプ結果表示アドレスは `FullAddr(6桁):Bank(2桁):Ofst(4桁)` とする
- `FullAddr` は `Window サイズ × Bank + Window 内オフセット` とする
- `Ofst` は `Window` 先頭からのオフセットとする

`Mapper` 表示時は以下の手順でダンプする。

1. `SwitchAddr` に `Bank` の値を書き込む
2. `Window` 領域をダンプする
3. ダンプした `Window` サイズ分のデータだけを HEX 表示する

- `ASCII 8K`、`ASCII 16K`、`Konami`、`Custom` のいずれも同じ流れで扱う
- `MapperType` は `Window` と `SwitchAddr` の既定値を与えるプリセットとする

### 1-2. Mapper ダンプ時の進捗表示

- Mapper ダンプは時間がかかるため、ステータスバーにプログレスバーを表示する
- 進捗は現在読み出している `Window` サイズ分の転送進行が分かること
- プログレスバーは Mapper ダンプ完了時に消す、または完了状態にする

### 1-3. Mapper 表示時の 1 バイト編集

- Mapper 表示時も HEX 編集を許可する
- 1 バイト編集時は、表示中の `Bank` と `Ofst` に対応する 1 バイトを書き換える
- 書込み対象アドレスは `Window` の先頭アドレス + 表示上のオフセットとする
- 1 バイト編集時は以下の手順で書き込む

1. `SwitchAddr` に対象バンク番号を書き込む
2. `Window` のオフセットアドレスにデータを 1 バイト書き込む
3. 該当バンクのみを読み直して表示バッファへ反映する

- 1 バイト編集後は全バンクを再読込しない
- 1 バイト編集後の再読込対象は、編集した該当バンクのみとする

### 2. 手動実行ウィンドウ

- `デバッグ -> 手動実行` を選ぶと手動実行ウィンドウを開く
- 手動実行ウィンドウに `Command` 入力欄と実行ログ欄を配置する
- 手動実行ウィンドウのログには、そのウィンドウから実行したコマンド結果のみを表示する
- メイン画面での起動時アクセスログや自動読出しログは表示しない

### 3. アクセスログダイアログ

- `デバッグ -> ログ` を選ぶとアクセスログを別ダイアログで開く
- アクセスログダイアログには adapter との送受信ログを表示する
- メイン画面には `Command` と `Log` を配置しない

## 通信仕様の扱い方

### コマンド送信

- GUI で入力された 1 行をそのまま送信する
- 改行は `CR`, `LF`, `CRLF` の受け側仕様差異を吸収できるようにする
- 送信後は応答をログへ表示する

### 応答受信

- 文字列応答は行単位で受信し、ログ欄へ表示する
- `OK` / `FAIL` は結果判定として扱う
- `BSND` などのバイナリ応答は文字列と混在するため、バイナリ受信モードを分けて扱う

### バイナリ転送

以下は「コマンド送信後に固定長のバイナリ本体を送る / 受ける」処理として実装する。

- `BRCV`
- `BSND`
- `SMTR`
- `SMTW`
- `IOTR`
- `IOTW`

実装上は以下を分離する。

- コマンド入力層
- バイナリ入出力層
- 画面表示層

## 実装上の注意

### 1. UI のフリーズ防止

シリアル待ち受けは GUI スレッドで直接行わず、別スレッドか `after()` ベースでポーリングする。

### 2. 送受信の競合回避

- 送信中に別コマンドを重ねない
- バイナリ転送中は入力欄を一時的に無効化する
- 応答待ちタイムアウトを設ける

### 3. ログの可読性

- 送信行は `>` プレフィックス
- 受信行は `<` プレフィックス
- バイナリは HEX 表示とファイル保存を切り替えられるようにする

### 4. コマンドの前処理

- 大文字小文字を気にせず入力できるようにする
- 16進数入力の補助を入れる
- 省略可能引数は GUI 側で空欄を許可する

### 5. 初期読出しの失敗時処理

- `HVER` に失敗した場合、以降の起動処理は行わない
- エラーは画面上で明示する
- 再試行の要否は初期段階では未定義とする

## 実装順

### 1. 起動と読出し

- `--device` によるシリアルデバイス指定
- `HVER` による起動時疎通確認
- 64kB 全体の読出し
- HEX エディタ風の表示
- 手入力コマンド送信
- ログ表示

### 2. HEX編集

以下をまとめて「HEX編集」タスクとする。

#### 2-1. HEX 編集の基盤

- 表示専用の HEX 領域を編集可能な構造へ切り替える
- カーソル位置とバッファ位置を相互に対応付ける
- HEX 部分だけ編集可能にする
- アドレス列とキャラクタ列は直接編集不可とする
- `0-9`, `a-f`, `A-F` 以外の入力は無視する
- 変更バイトは赤表示にする

#### 2-2. 編集状態の管理

- 元データ 64kB と編集中データ 64kB を分けて保持する
- 変更済みバイト集合を管理する
- 未保存変更の有無を判定できるようにする

#### 2-3. キャラクタ列の連動更新

- HEX 編集時に右側のキャラクタ表示を即時更新する
- ASCII 列からの直接編集は行わない

### 3. 次タスク

以下は「HEX編集」の後に進める別タスクとする。

- 明示操作によるデバイスへの一括書き戻し
- `SCHK`、`HVER` などの補助操作
- 手動コマンド履歴
- 編集ナビゲーション

## 依存ライブラリの提案

最小構成では以下で十分。

- `pyserial`

必要になった場合のみ追加する候補。

- `pyinstaller`:
  配布ビルド用
- `ttkbootstrap` などのテーマ系:
  見た目を整える用途。ただし依存増加があるため初期段階では非推奨

## 実装方針の結論

- GUI は `tkinter`
- 通信は `pyserial`
- 余計な外部依存は増やさない
- まずは「接続・送信・受信ログ・基本コマンド」の最小実装から始める
